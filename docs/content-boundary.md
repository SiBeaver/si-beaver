# 内容边界规范：代码仓库 vs sibs

本文档定义了项目信息的存放边界。当项目接入 sibs（si-beaver MCP）后，应遵循以下原则决定信息归属。

## 核心原则

| 归属 | 判断标准 |
|------|----------|
| **代码仓库** | 描述"怎么写、怎么构建、怎么跑" |
| **sibs** | 描述"为什么做、做什么、学到了什么、有什么风险" |

换言之：**实现细节跟代码走，认知过程放 sibs。**

## 代码仓库应保留的内容

| 类型 | 示例 | 原因 |
|------|------|------|
| 技术选型说明 | `docs/tech-stack.md` | 开发者入手需要 |
| API 接口文档 | OpenAPI spec / 代码注释 | 与代码同步演进 |
| UI/组件设计稿 | `docs/design/` | 前端开发参考 |
| 构建与部署配置 | Dockerfile, CI 配置 | 构建流程是代码的一部分 |
| README | 项目简介、快速启动 | 仓库入口 |
| CONTRIBUTING | 贡献指南 | 协作规范 |

## sibs 应存储的内容

| 认知类型 | 示例 | 对应节点类型 |
|----------|------|-------------|
| 目标与需求 | "MVP 要支持多项目" | Goal |
| 决策记录 | "选 SQLite 而不是 Postgres 因为..." | Decision |
| 探索过程 | "试了 X 方案，发现 Y 问题" | Exploration |
| 风险与技术债 | "当前无鉴权，部署时需加反代" | Risk |
| 学到的知识 | "Hono 的中间件顺序影响 CORS" | Knowledge |
| 路线图与阶段目标 | "26.6 目标：多租户" | Goal |
| 运维与环境说明 | "部署在 10.1.1.40:7420" | Knowledge |

## 灰色地带处理

有些内容两边都有价值，遵循以下规则：

1. **技术选型的"理由"部分** — 代码里保留结论（选了什么），sibs 记录推理过程（为什么选、排除了什么）
2. **架构变更** — 代码里体现最终结构，sibs 记录变更动机和权衡
3. **Bug 修复** — 代码里是 fix 本身，sibs 记录根因分析（如果有通用价值）

## 给 AI 工具的指引

当 AI 工具通过 MCP 连接 sibs 时：

- **写代码前**：查询 sibs 获取目标、决策、已知风险，避免重复探索
- **写代码时**：技术实现细节写在代码注释里，不要推到 sibs
- **写完代码后**：将有长期价值的决策、发现、风险推送到 sibs
- **不要推送**：临时调试信息、具体代码片段、一次性操作记录

## 判断检查清单

在决定信息放哪里时，问自己：

- [ ] 删掉这个信息，代码还能正常构建和运行吗？→ 能 → sibs
- [ ] 这个信息 6 个月后还有参考价值吗？→ 有 → sibs
- [ ] 换一个开发者来，他需要这个信息才能改代码吗？→ 是 → 代码仓库
- [ ] 这个信息描述的是一次性的实现，还是可复用的认知？→ 认知 → sibs
